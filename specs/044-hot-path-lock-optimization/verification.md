# Verification Ledger: Hot Path Lock Optimization

This ledger records task-local proof for the hot-path lock optimization work. T002 creates the structure only; later tasks append or update rows as they add expected-red tests, run focused commands, and capture final results.

## Expected-Red Proof Strategy

Expected-red guard-release tests should prefer observable reentrant callback or reentrant operation behavior: the test should arrange a callback, sampler, delivery path, or dispatch continuation that calls back into the relevant API and would deadlock, block progress, or return the wrong result if the original guard were still held. This makes the proof about externally visible behavior instead of only internal instrumentation.

`OPCUA_TRACE_LOCKS` lock tracing is secondary/supporting evidence. Use trace assertions to corroborate guard release, or as the primary proof only when the behavior cannot be observed directly without changing the public contract.

Every proof must preserve OPC UA externally visible behavior and status semantics. Guard narrowing is accepted only when Read, Write, Call, Publish acknowledgement, MonitoredItem queueing, Session, SecureChannel, and PubSub outcomes remain equivalent to the pre-change behavior required by the referenced OPC UA sections.
