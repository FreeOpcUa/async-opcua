# OPC UA Part 6 v1.05.07 Conformance Findings

Scope: OPC UA Binary §5.2, §5.3 subtype/XML-adjacent rules where relevant, OPC UA JSON §5.4, MessageChunk/message security §6.7, UACP Hello/Ack/Error §7.1. Spec text in `/tmp/part6.txt` is treated as authoritative.

## Candidate Divergences

1. Boolean binary decoding treats only byte `1` as true. Spec says decoders shall treat any non-zero byte as true (Part 6 lines 1540-1544). Code returns `read_u8(stream)? == 1`, so bytes `2..255` decode as false (`async-opcua-types/src/basic_types.rs:41`).
2. Variant binary decoding rejects reserved built-in type IDs 26-31. Spec says decoders shall accept IDs 26-31, assume ByteString or array of ByteStrings, and pass them to the application (lines 1999-2008). Code maps only known IDs 1-25 and returns `None`/`BadDecodingError` for other values (`async-opcua-types/src/variant/type_id.rs:232`, `async-opcua-types/src/variant/mod.rs:400`, `async-opcua-types/src/variant/mod.rs:433`, `async-opcua-types/src/variant/mod.rs:603`).
3. DataValue picoseconds are not clamped to 9999. Spec says picosecond fields shall contain values less than 10000 and decoders shall treat values >=10000 as 9999 (lines 2077-2080). Code decodes `u16` and preserves it when the matching timestamp is present (`async-opcua-types/src/data_value.rs:227`, `async-opcua-types/src/data_value.rs:238`, `async-opcua-types/src/data_value.rs:243`).
4. JSON Int64/UInt64 encode/decode uses JSON numbers. Spec says Int64 and UInt64 shall be formatted as decimal numbers encoded as JSON strings (lines 3032-3037). Code applies the generic JSON number implementation to `u64` and `i64` (`async-opcua-types/src/json.rs:208`, `async-opcua-types/src/json.rs:235`, `async-opcua-types/src/json.rs:239`).
5. JSON NodeId and ExpandedNodeId use object forms instead of Part 6 v1.05.07 strings. Spec says NodeId and ExpandedNodeId values shall be encoded as JSON strings using the 5.1.12 format (lines 3087-3098, 3101-3118). Code begins JSON objects and emits fields such as `Id`, `IdType`, `Namespace`, and `ServerUri` (`async-opcua-types/src/node_id/json.rs:34`, `async-opcua-types/src/node_id/json.rs:40`, `async-opcua-types/src/expanded_node_id.rs:82`, `async-opcua-types/src/expanded_node_id.rs:88`).
6. JSON Variant uses `Type`/`Body`, not `UaType`/`Value`. Spec defines Variant JSON fields as `UaType`, `Value`, and `Dimensions` (lines 3242-3259). Code emits/consumes `Type`, `Body`, and `Dimensions` (`async-opcua-types/src/variant/json.rs:77`, `async-opcua-types/src/variant/json.rs:79`, `async-opcua-types/src/variant/json.rs:89`, `async-opcua-types/src/variant/json.rs:193`).
7. JSON ExtensionObject emits `UaBody` for JSON-encoded structures and encodes null as JSON null. Spec says JSON-encoded structures are encoded as JSON objects with `UaTypeId` inserted, while `UaEncoding`/`UaBody` are only used for UA Binary/XML encoded Structures; the default ExtensionObject is an empty object in VerboseEncoding and omitted in CompactEncoding (lines 3202-3240). Code emits `null` for no body and always writes `UaBody` for non-null bodies (`async-opcua-types/src/extension_object.rs:264`, `async-opcua-types/src/extension_object.rs:270`, `async-opcua-types/src/extension_object.rs:283`, `async-opcua-types/src/extension_object.rs:300`).
8. JSON ExtensionObject duplicate-name rejection is not evident. Spec says decoders shall report decoding errors if a JSON object has multiple fields with the same name (lines 3213-3214). Code iterates `next_name()` and overwrites `type_id`, `encoding`, `raw_body`, and `body` without tracking prior field names (`async-opcua-types/src/extension_object.rs:319`, `async-opcua-types/src/extension_object.rs:327`, `async-opcua-types/src/extension_object.rs:329`).
9. Asymmetric security header does not enforce `SecurityPolicyUriLength <= 255`. Spec says the URI length shall not exceed 255 and invalid security header lengths require closing the communication channel (lines 4222-4228, 4271-4272). Code decodes it as a general `UAString` with the global string limit and does not apply a 255-byte URI limit (`async-opcua-core/src/comms/security_header.rs:137`, `async-opcua-core/src/comms/security_header.rs:142`; global limit at `async-opcua-types/src/lib.rs:49`).
10. Hello buffer-size validation always requires 8192 bytes. Spec allows 1024-byte Send/ReceiveBufferSize if the sender intends to use an ECC SecurityPolicy and requires 8192 otherwise (lines 5262-5267). Code rejects any HEL buffer below `MIN_CHUNK_SIZE` 8192 without considering ECC intent (`async-opcua-core/src/comms/tcp_types.rs:40`, `async-opcua-core/src/comms/tcp_types.rs:340`).
11. Negotiated unlimited inbound chunk count is converted to one chunk when both max chunk count and max message size are zero. Spec says MaxChunkCount value zero indicates no limit (lines 5275-5279, 5325-5329). Server code computes an effective limit of `1` when both limits are zero, then rejects once `pending_chunks.len() >= max_chunks` (`async-opcua-server/src/transport/tcp.rs:116`, `async-opcua-server/src/transport/tcp.rs:482`).
