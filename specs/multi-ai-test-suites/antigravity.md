# Antigravity (Gemini 3.5) Independent Coverage Proposal

Generated by delegating an independent test-coverage cross-check to Google Antigravity.
It inspected the repo (it cites specific files/lines) and proposed 11 novel cases.

## Priority 1 — Concurrency, handshakes, security boundaries

### 1. SecureChannel renewal token overlap & expiry
Server must accept BOTH old and new SecurityTokenId during the renewal grace window, then
reject the old token after its lifetime+grace expires. Part 6 §6.7.4 / Part 4 §5.5.2.
Impl: Rust integration test, low channel lifetime (~2s), renew while sending old-token requests.

### 2. Session hijack / cross-channel transfer
Create a session on Channel A (cert A); send ActivateSession for that token over Channel B
(cert B). Must be rejected (BadSecurityChecksFailed / BadSecureChannelIdInvalid). Directly tests
`manager.rs` `is_cross_channel_transfer_forbidden` / `is_client_certificate_channel_mismatch`.
Part 4 §5.6.3. Impl: raw service builders, two channels.

### 3. X509 / identity-token signature tampering & wrong-key verification
Tamper the user signature in ActivateSession; cert with no digital-signature KeyUsage; revoked
user cert. Catches auth bypass if the signature is verified against the wrong public key.
Part 4 §5.6.3 / §7.41.3. Impl: raw ActivateSessionRequest with forged client_signature.

### 4. Slow-loris / half-open handshake DoS (timing, not malformed input)
Send HEL, get ACK, then dribble OPN byte-by-byte / leave hundreds of connections half-open.
Fuzzing tests malformed bytes, not timing exhaustion. Server must time out half-open conns and
stay responsive. Part 6 §7.1.2 / Part 2. Impl: raw scripted TCP client.

## Priority 2 — Subscription state machine & address-space interactions

### 5. TransferSubscriptions old-session StatusChangeNotification  [VERIFY — likely COVERED]
Antigravity cites AUDIT.md Gap #2. NOTE: `transfer_subscriptions_notifies_old_session`
(subscriptions.rs) already asserts Good_SubscriptionTransferred to the old session. Cross-check
self-correction candidate.

### 6. Dynamic EURange update on Percent deadband  [HIGH — POTENTIAL BUG]
`monitored_item.rs:~837` caches `eu_range` at creation and only refreshes on item modify. If a
client rewrites the variable's EURange child node mid-life, the percent-deadband filter keeps
using the stale range → missed/excess notifications. Part 8 §6.2.2 / Part 4 §7.22.
Impl: AnalogItem EURange [0,100], 5% deadband; write EURange [0,50]; verify the deadband adapts.

### 7. MonitoredItem on a deleted node  [HIGH — PANIC RISK]
With writable address space, delete a node that has an active monitored item. Server should emit
a Bad_NodeIdUnknown notification and not panic (unwrap on missing node). Part 4 §5.7.3 / §5.13.1.

### 8. EventQueueOverflowEventType generation  [VERIFY — likely COVERED]
Cites 029 STATUS.md Gap D. NOTE: `event_queue_overflow_inserts_overflow_event` (subscriptions.rs)
already exists. Cross-check self-correction candidate; could still expand SelectClause filtering.

## Priority 3 — Robustness & parser boundaries

### 9. NumericRange OOB / rank-mismatch on multi-dim arrays  [PARTLY COVERED by fuzz]
3D array, read range "0:2,0:4" (rank mismatch) / "0:2,0:1,1:5" (OOB); write 1D into 2D range.
Must return Bad_IndexRangeInvalid/NoData, not panic. Part 4 §7.27. (Feature 017 fuzzed substring
+ extent underflow; explicit rank-mismatch cases may add value.)

### 10. NodeManagement circular references → recursive-browse stack overflow  [HIGH — DoS PANIC]
AddNodes/AddReferences to create A→B→A hierarchical loop, then recursive Browse /
TranslateBrowsePaths. Server must bound depth / track visited, not stack-overflow. Part 3 §5.2 /
Part 4 §5.7.

### 11. Chunk reordering / duplication / fragmented-message reassembly  [overlaps Codex P0]
Force fragmentation (small max chunk), MITM reorders/duplicates chunks of a large Write. Server
must reject (BadSecurityChecksFailed/BadSequenceNumberInvalid) and free buffers. Part 6 §6.7.2/3.
